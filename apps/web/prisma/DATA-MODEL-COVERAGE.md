## Section MT5 (preuves & réconciliation)

Chaîne complète de la **preuve par la réalité** (brief 03 §29-§33) :

- **Import preuve** : `Mt5AccountProof` (schema:2426) — `fileKey`, `fileHash` (SHA-256 anti-double-upload, `@@unique([memberId,fileHash])`), `accountType`, `ocrStatus` (pending/done/failed), `claudeRunId` (schema:2451), `uploadedAt`.
- **Détection comptes** : `BrokerAccount` (schema:2386) — `type` (prop_firm/personal), `accountLogin` (clé de dédup « combien de comptes »), `detectedByAI`, `confidence` ; dénormalisé sur `User.detectedAccountCount` (schema:297).
- **Réalité extraite (vision)** : `ExtractedPosition` (schema:2475) — `source=mt5_screen_ocr`, `ticket`, `symbol`, `side`, `openTime/closeTime`, `volume`, `entryPrice/exitPrice`, `pnl`, `confidence`, `proofId` (provenance forensique).
- **Confrontation** : `Trade.source`/`verifiedAt`/`matchStatus` (schema:654) + `Discrepancy` (schema:2530) — `type` (missing_declared/false_declared/mismatch/unfilled_no_reason/meeting_missed_no_reason), `severity`, `status`, `declaredTradeId`, `extractedPositionId`, `claudeReasoning` (§2-clean), `memberReason`.

## Section Scores (4 dimensions + constance)

- **4 dimensions déclaratives** — `BehavioralScore` (schema:1264) : `disciplineScore`, `emotionalStabilityScore`, `consistencyScore`, `engagementScore` (schema:1275-1278), `Int?` = `insufficient_data` ; transparence via `components` Json (parts par dimension) + `sampleSize` + `windowDays`.
- **Score de constance reality-confronted** (modèle DÉDIÉ, distinct des 4 ci-dessus car confronté à la réalité externe) — `ConstancyScore` (schema:2595) : `value` (Float 0-100), `breakdown` Json `{honesty, regularity, discipline}` (schema:2605), `periodStart/periodEnd`, `@@unique([memberId,periodStart])`.
- **Présence (partial)** : pas de colonne `presenceScore` nommée — la présence aux réunions alimente la dimension **engagement** (`BehavioralScore.components.engagement.parts` + primitive `countMeetingAttendance`). Couverture **indirecte/agrégée**, assumée par design ; ce n'est **pas** un axe orphelin (les données de présence sont intégralement persistées — voir axe « présence aux réunions » : `Meeting`/`MeetingAttendance`).

## Section Event-log horodaté

- `AuditLog` (schema:486) — `action`, `metadata` Json, `createdAt`, `@@index([userId,createdAt])` + `@@index([action,createdAt])` : journal d'événements RGPD horodaté global.
- `ScoreEvent` (schema:2625) — `delta`, `reason` (filled/forgot_no_reason/reality_gap/false_declaration), `relatedDiscrepancyId`, `createdAt`, `@@index([memberId,createdAt desc])` : journal event-sourced expliquant chaque variation de constance. **Contrat append-only** : voir [`../src/lib/audit/APPEND-ONLY.md`](../src/lib/audit/APPEND-ONLY.md).
- **Versionnage léger (pas d'écrasement)** : snapshots datés `BehavioralScore @@unique([userId,date])`, `ConstancyScore @@unique([memberId,periodStart])`, `WeeklyReport @@unique([userId,weekStart])` → courbes d'évolution / lecture « où en était le membre à telle date ».
- **Validité longitudinale** : `instrumentVersion` sur `MindsetCheck`, `WeeklyScheduleQuestionnaire`, `OnboardingInterview`. Chaque modèle porte `createdAt` (+ `updatedAt @updatedAt`).

## Conclusion — absence d'axe orphelin

Les 21 axes du CONTEXTE GLOBAL et des tâches §2 (sessions 1→10) sont tous mappés sur des modèles/champs réels : **0 axe orphelin**. Le seul `partial` (score de _présence_) est une couverture indirecte **par construction** — toutes les données de présence sont persistées (`Meeting`/`MeetingAttendance`) et nourrissent la dimension `engagement` ; elles ne manquent nulle part, elles ne sont simplement pas exposées comme une 4ᵉ dimension de score nommée. **DoD#2 (couverture 100 % champ par champ, MT5 + scores inclus) est satisfait.**

> Le langage commun de ces métriques (clé, libellé, axe, unité, poids) est figé dans le registre central [`../src/lib/metrics/registry.ts`](../src/lib/metrics/registry.ts) (verrouillé par `registry.test.ts`).
